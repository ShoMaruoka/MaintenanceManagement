# Spec: 本番前準備(Prepare)画面 - 横並び比較機能

## 対象Issue
GitHub Issue #14「本番前準備 / prepareの比較レイアウトを横並びにしてほしい」

## 目的 (Objective)

現在の本番前準備画面（`frontend/src/pages/PrepareForPrd.tsx`）は、各DB（kaios / gos / paf / duskin）ごとに独立したカードでファイル一覧（今回適用する / 保留中）を表示している。DBごとにカードが分かれているため、「このファイルは他のサイトにも存在するか」「サイトごとに適用漏れがないか」を確認する際、カードを目で行き来する必要があり手間がかかる。

本機能では、既存のカード表示はそのまま維持しつつ、「比較」ボタンを押すことで画面内に**表形式の横並び比較ビュー**を表示できるようにする。表は行=ファイル名、列=DBとし、各DBでの有無を一目で確認できるようにする。さらに、比較結果をクリップボードコピー／テキストファイルとしてダウンロードできるようにし、WinMergeなど外部ツールでの比較にも対応する。

**ユーザー:** 本番反映作業を行う保守担当者。複数サイト（DB）へ同一のSQLファイル群を展開する際、サイト間の適用漏れ・差分を確認する用途で使う。

**成功の定義:**
- 比較ボタン押下で、カード一覧 ⇔ 比較表を画面内でトグルできる。
- 比較表で、ファイル名ごとに各DB（kaios/gos/paf/duskin）に存在するかどうかが列で分かる。
- 全DB共通ファイルと一部DB固有ファイルが視覚的に色分けされている。
- 比較表の内容をクリップボードにコピーでき、また `.txt` としてダウンロードできる。
- 対象は「今回適用する(deployed)」「保留中(hold)」の両方。

## スコープ外 (Out of Scope)

- Issueの「ベスト要望」であるGit Log差分比較機能・WinMerge直接連携機能は今回のスコープに含めない（将来の別issueとする）。
- ファイル内容(SQL本文)同士の差分表示は行わない。比較はファイルの**有無**のみを対象とする。

## Tech Stack

- Frontend: React 18 + TypeScript + Vite（`frontend/`）
- Backend: ASP.NET Core (.NET) （`backend/`) ※本機能は既存API (`GET /api/prepare/files`) のレスポンスをフロントエンドのみで加工するため、バックエンド変更は不要と想定。
- スタイリングは既存同様、`index.css` にクラスベースで追記（CSS-in-JSやライブラリ導入はしない）。

## 機能詳細

### 1. 表示切り替え
- 既存カード一覧の上部（説明文の直下、または `prep-grid` の直上）に「比較」ボタンを追加。
- クリックすると `prep-grid`（カード表示）が非表示になり、比較表ビューに置き換わる。もう一度押す（ボタンラベルは「一覧に戻る」等に変化）とカード表示に戻る。
- 表示状態はコンポーネント内 `useState<'cards' | 'compare'>` 等で管理。ページ遷移では保持しなくてよい。

### 2. 比較表のデータ構造
- 対象は `dbEntries` の全ファイル（`source: 'deployed' | 'hold'` 両方）。
- 行キーは「今回適用する」「保留中」のセクションごとに分け、セクション内でファイル名（+ dbType）をユニーク行とする。
  - 行の一意性: 同名ファイルでも `dbType`（sqlserver/mariadb）が異なれば別行として扱う。
- 列は DB（kaios, gos, paf, duskin）の固定4列。
- セルには対象DBに当該ファイルが存在するかを ✓ 等のマークで表示する。チェック状態（適用/保留）も反映する場合は、既存の checked セットを参照してチェックマークの見た目に反映してよい（クリックでの再選択は比較ビュー内では不要、閲覧用途に限定）。

### 3. 差分の視覚的強調
- 全DB（4件）に存在するファイル行 → 通常表示（既存の文字色）。
- 一部のDBにしか存在しないファイル行 → 背景色や左ボーダー等で色分け強調（例: 淡いオレンジ背景）。
- 「今回適用する」「保留中」で表（またはセクション見出し付きの表ブロック）を分ける。カード表示のセクション構成に合わせ、比較ビューでも「今回適用する」表 → 「保留中」表の順に並べる。
- 保留中セクションの各セルには、存在有無に加えて現在のチェック状態（適用予定に切り替え済みかどうか）も視覚的に反映する（例: ✓マークの色や補助アイコンで「保留のまま」と「今回適用予定に変更済み」を区別）。閲覧専用でよく、比較ビュー内でのチェック操作（トグル）は不要。

### 4. テキストエクスポート
- 比較ビュー内に「コピー」ボタンと「ダウンロード」ボタンを設置。
- テキスト形式: タブ区切り（TSV）。セクション（今回適用する/保留中）ごとに見出し行 + ヘッダー行（`ファイル名\tkaios\tgos\tpaf\tduskin`）を出力し、値は存在すれば `○`、なければ空欄とする。保留中セクションでは、チェック済み（適用予定）を `○(適用予定)` のように区別可能な表記とする（具体的な記号は実装時に確定）。
- 「コピー」: `navigator.clipboard.writeText()` でクリップボードにコピーし、コピー完了のフィードバック（一時的なラベル変化等）を表示する。
- 「ダウンロード」: 生成したテキストを Blob 化し `<a download>` で `.txt` ファイルとしてダウンロードする（ファイル名例: `prepare-compare_YYYYMMDD_HHmmss.txt`）。

## テスト戦略

現状このリポジトリに自動テスト基盤（Jest/Vitest等）は導入されていない。本機能追加でも新たにテストフレームワークを導入するかは実装フェーズで判断する（デフォルトは導入しない・手動確認とする）。

- 手動確認: `npm run dev` でフロントエンドを起動し、本番前準備画面で
  - 比較ボタンでの表示切り替え
  - DB間でファイルの有無が正しく表示されること
  - 差分色分けの見た目
  - コピー/ダウンロードの動作
  を目視確認する。
- ビルド確認: `npm run build`（`tsc` の型チェックを含む）がエラーなく通ること。

## Boundaries

- **常に行うこと:** 既存のカード表示・選択（チェックボックス）機能や実行フローを壊さないこと。`npm run build` の型チェックを通すこと。
- **確認してから行うこと:** 新規npm依存の追加、バックエンドAPIの変更・追加、比較表のデータ構造やエクスポート形式を大きく変える場合。
- **行わないこと:** Git Log比較機能やWinMerge直接連携の実装（スコープ外）。ファイル内容(SQL本文)の差分表示。既存の本番適用実行ロジック(`runPreparation`, `FastCopyService`等)の変更。

## Success Criteria

- [ ] 「比較」ボタンでカード表示 ⇔ 比較表表示を切り替えられる
- [ ] 比較表で各DB列のファイル有無が一目でわかる
- [ ] 全DB共通ファイルと一部DB固有ファイルが色分けされている
- [ ] 比較表が「今回適用する」「保留中」のセクションに分かれている
- [ ] 保留中ファイルのチェック状態（適用予定への変更）が比較表で識別できる
- [ ] 比較結果をTSV形式でクリップボードにコピーできる
- [ ] 比較結果を `.txt`（TSV形式）としてダウンロードできる
- [ ] `npm run build` がエラーなく通る
- [ ] 既存のファイル選択・本番前準備実行フローに影響がない

## Open Questions

なし（下記の通り確定済み）:
- 比較表は「今回適用する」「保留中」でセクション分けする。
- テキストエクスポートはTSV表形式（セクション見出し + ヘッダー行 + 各DB列）。
- 保留中ファイルの適用チェック状態（適用予定への変更有無）も比較表・エクスポートに反映する。
